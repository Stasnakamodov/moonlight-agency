# Задача: Починить спиральную камеру в StarField.tsx

## Файл: `src/components/effects/StarField.tsx`

## Что есть сейчас

Камера спирального спуска (`helixPosition`, строка ~343) управляет движением звёздного фона при скролле страницы. Параметры:

```
DESCENT_RANGE = 8500    — дальность линейного спуска
HELIX_TURNS   = 2.5     — кол-во оборотов спирали
HELIX_RADIUS  = 2000    — амплитуда осцилляции
DIAG  = (-0.707, -0.707) — направление спуска (ВВЕРХ-влево — это БАГ, canvas Y+ = вниз)
PERP  = (-0.707, +0.707) — перпендикулярная ось для вращения
```

Камера делает `mainX/Y = t * DESCENT_RANGE * DIAG` (линейный спуск) + `perpOsc * PERP + diagOsc * DIAG` (спиральная осцилляция).

## Проблема глазами пользователя

**Ключевое правило: при скролле ВНИЗ ВСЁ должно ВСЕГДА двигаться вниз — камера, звёзды, космос, фон. Ничего никогда не должно ехать вверх, если пользователь крутит колёсико вниз. Это касается ВСЕХ слоёв: пыль, далёкие звёзды, близкие звёзды, MW-фон, туманности — ВСЁ крутится вниз.**

Ощущение — винтовой лифт. Ты едешь вниз, стенки шахты закручиваются вокруг тебя. Всё вокруг непрерывно уходит вверх (обратно направлению спуска). Разные слои с разной скоростью — это и создаёт спиральный параллакс.

**Сейчас сломано**: осцилляция (HELIX_RADIUS=2000) перебивает линейный спуск. В каждом витке есть участки, где камера **едет назад вверх**, хотя юзер скроллит вниз. Один оборот вниз, потом кидает назад вверх, потом опять вниз — пилообразная тряска вместо плавного спуска. Звёзды и фон начинают ехать в обратную сторону.

**Нужно**: при скролле вниз ВСЁ ВСЕГДА идёт вниз — непрерывно, без пауз, без разворотов. Спиральное вращение — это боковое покачивание (как стенки лифтовой шахты проносятся мимо), НЕ вертикальная тряска. Количество оборотов должно быть достаточным чтобы спираль ощущалась на протяжении ВСЕГО скролла (страница длинная, 5 зон). Спуск непрерывный — никаких участков где он "замирает" или тем более разворачивается.

## Математический контекст

Условие "камера всегда идёт вниз": скорость Y-компонента линейного спуска должна ВСЕГДА превышать максимальную Y-компоненту осцилляции.

```
DESCENT_RANGE * |DIAG_Y| > HELIX_RADIUS * 2π * HELIX_TURNS * |PERP_Y|
```

Если PERP имеет Y-компонент (сейчас PERP_Y=0.707), то осцилляция напрямую воюет со спуском по Y. Чем больше HELIX_TURNS и HELIX_RADIUS — тем сильнее осцилляция перебивает спуск.

## Также есть баг направления

`DIAG_Y = -0.707` — это ВВЕРХ в canvas (Y+ = вниз). Нужно `DIAG_Y = +0.707` для спуска вниз-влево.

После исправления DIAG на `(-0.707, +0.707)`, текущий PERP `(-0.707, +0.707)` совпадёт с DIAG. Правильный перпендикуляр (90° CW): `(+0.707, +0.707)`.

## Что нужно сделать

1. **Исправить DIAG_Y**: `-0.707` → `+0.707`
2. **Пересчитать PERP**: `(+0.707, +0.707)` — 90° CW от нового DIAG
3. **Подобрать HELIX_TURNS, HELIX_RADIUS, DESCENT_RANGE** так чтобы:
   - Камера Y ВСЕГДА монотонно растёт (скроллим вниз = спускаемся)
   - Спиральное вращение ощутимо на всём протяжении скролла
   - Нет участков "стопора" где спуск почти останавливается (min dY/dt не менее ~15-20% от среднего)
4. **Тайлинг текстур**: MW-слои, зонные туманности и deep field используют `drawWrapped()` для тайлинга, но текстуры НЕ бесшовные (виньетка, шумовая структура MW-полосы). При большом сдвиге камеры видны швы. Решение: либо overscan (генерировать текстуры больше viewport), либо сделать шум тайлящимся.

## Верификация

Перед тем как писать код — выведи таблицу `helixPosition(t)` для t=0, 0.05, 0.10, ..., 1.0 и покажи:
- camX, camY на каждом шаге
- deltaY между шагами (ДОЛЖЕН быть > 0 ВСЕГДА)
- min deltaY / max deltaY (отношение — чем ближе к 1, тем равномернее спуск)

Пользователь должен утвердить числа ПЕРЕД тем как ты начнёшь редактировать файл.
